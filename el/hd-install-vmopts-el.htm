Εικονικές μηχανές

  • KVM για Intel VT ή AMD-V
  • Virtualbox
  • QEMU
  • Installing other distributions to an image

The following examples use aptosid, simply substitute aptosid with the distribution of your choice.

Installing other distributions to a VM image

Note: If and when you wish to install to a virtual machine image, most Linux distributions will probably only need an allocation of 12G. However if you have a requirement to have MS Windows in a virtual machine, you will need to allocate about a 30G, or more, to the image. All image allocation sizes ultimately depend on your requirements.

Generally, an image allocation size, will not take up additional hard drive space until data is installed. Even then, it will only take space dynamically on a hard drive, contingent to the actual amount of data that expands on the image. This is due to the compression ratios of qcow2.

Enabling a KVM Virtual Machine

KVM is a full virtualisation solution for Linux running on x86 CPU with virtualisation extensions (Intel VT or AMD-V).

Prerequisites

To ascertain whether your hardware supports KVM, ensure that KVM is enabled in the BIOS, (in some cases on an Intel VT or AMD-V system it may not evident as to where the switch is, therefore assume that it is in a KVM state). The way to check is in a console run:

cat /proc/cpuinfo | egrep --color=always 'vmx|svm'

If you see svm or vmx in the cpu flags field, your system supports KVM. (Otherwise go back to the BIOS if you believe it is supported and check again, else search the internet as to where in the BIOS menus KVM 'enable' could be hiding).

Should your BIOS not support KVM refer to Virtualbox or QEMU

To install and run KVM, firstly ensure that Virtualbox modules are not loaded, (--purge them is the best option), then depending on your chipset:

For vmx:

apt-get install qemu-kvm qemu-utils
modprobe kvm-intel

For svm:

apt-get install qemu-kvm qemu-utils
modprobe kvm-amd

When you start your system the qemu-kvm initscripts will take care of loading the modules.

Using KVM to boot an aptosid-*.iso

For 64 bit as user:

$ QEMU_AUDIO_DRV=alsa qemu-system-x86_64 -machine accel=kvm:tcg -net nic,model=virtio -net user -soundhw all -m 1024 -monitor stdio -cdrom <aptosid.iso>

For 32 bit replace qemu-system-x86_64 with qemu-system-i386.

Installing aptosid to a KVM image

First create a hard disk image, (this image will be minimal and only grow as required due to qcow2 compression ratios):


$ qemu-img create -f qcow2 aptosid-VM.img 12G

Boot the aptosid-*.iso with the following parameters to enable KVM to recognise that there is an QEMU hard disk image available:

$ QEMU_AUDIO_DRV=alsa qemu-system-x86_64 -machine accel=kvm:tcg -net nic,model=virtio -net user -soundhw all -m 1024 -monitor stdio -cdrom </path/to/aptosid-*.iso> -boot d </path/to/aptosid-VM.img>

For 32 bit replace qemu-system-x86_64 with qemu-system-i386.

Once the cdrom has booted up click on the aptosid installer icon to activate the installer, (or use the menu), click on the Partitioning tab and launch the partition application you prefer. For the partitioning you can follow the instructions from Partitioning the Hard Drive - Traditional, GPT and LVM (do not forget to add a swap partition If you are short of memory). Please be aware that formatting will take time so be patient.

You now have a VM ready for use:

$ QEMU_AUDIO_DRV=alsa qemu-system-x86_64 -machine accel=kvm:tcg -net nic,model=virtio -net user -soundhw all -m 1024 -monitor stdio -drive if=virtio,boot=on,file=<path/to/aptosid-VM.img>

Some guests do not support virtio, therefore you need to use other options when launching KVM, for example:

$ QEMU_AUDIO_DRV=alsa qemu-system-x86_64 -machine accel=kvm:tcg -net nic, -net user -soundhw all -m 1024 -monitor stdio -hda </path/to/your_guest.img > -cdrom your_other.iso -boot d

or

$ QEMU_AUDIO_DRV=alsa qemu-system-x86_64 -machine accel=kvm:tcg -net nic, -net user -soundhw all -m 1024 -monitor stdio -hda </path/to/your_guest.img>

See also: KVM documentation.

Managing your KVM virtual machine installations
apt-get install aqemu

When using AQEMU ensure that you choose the KVM mode from the drop down box for 'Emulator Type' in the 'General' tab. (Documentation for AQEMU is basically non existent, therefore a few 'trials by error' will be required to figure out the easy to use GUI, however a good start is to use first use the 'VM' menu followed by the 'General' tab).

Τα βήματα:

  • 1. δημιουργήστε ένα εικονικό αρχείο σκληρού δίσκου (hard disk image) για το VirtualBox
  • 2. εκκίνηστε από το ISO με VirtualBox
  • 3. εγκαταστήστε στον εικονικό σκληρό δίσκο

Απαιτήσεις

Συνιστώμενη ram: 1 giga: Ιδανική μνήμη 512 ΜΒ για τον επισκέπτη (guest) και 512 ΜΒ για τον κυρίως χρήστη (host). (μπορεί να τρέξει και με λιγότερη ram, αλλά μην αναμένετε να έχει καλή απόδωση).

Χωρητικότητα σκληρού δίσκου:Ενώ το VirtualBox από μόνο του έχει πολύ ελαφριά αποθήκευση (μια τυπική εγκατάσταση θα χρειαστεί περί τα 30 ΜΒ του σκληρού δίσκου ), οι εικονικές μηχανές θα απαιτήσουν αρκετά μεγαλύτερα αρχεία σε δίσκο για την αποθήκευση τους, στους σκληρούς δίσκους τους. Έτσι, για να εγκαταστήσετε το MS Windows XP (TM), παραδείγματος χάριν, θα χρειαστείτε ένα αρχείο που θα αυξηθεί εύκολα σε αρκετά GB. Για να έχετε aptosid σε VirtualBox πρέπει να διαθέσετε μια 5 giga εικονική κατάτμηση συν μια κατάτμηση swap.

Εγκατάσταση:

apt-get update
apt-get install virtualbox-ose-qt virtualbox-source dmakms module-assistant

Προετοιμασία του module:

m-a prepare
m-a a-i virtualbox-source

Μετά ενεργοποιήστε το Dynamic Module-Assistant Kernel Module Support για το virtualbox, έτσι ώστε την επόμενη φορά που θα αναβαθμιστεί ο πυρήνας Linux θα ετοιμαστεί ένα virtualbox module για αυτόν χωρίς να χρειαστεί να κάνετε κάτι εσείς. Για να το κάνετε αυτό, προσθέστε virtualbox-source στο /etc/default/dmakms αρχείο ρυθμίσεων. 

mcedit /etc/default/dmakms
virtualbox-source

Μετά επανεκκινήστε τον υπολογιστή.

Ένα Qt GUI εργαλείο είναι διαθέσιμο για να σας βοηθήσει με την ρύθμιση του QEMU:

apt-get install qtemu

Δημιουργία της εικόνας σκληρού δίσκου

Για να τρέξετε το qemu θα χρειαστείτε πιθανώς μια εικόνα σκληρού δίσκου. Αυτό είναι ένα αρχείο που αποθηκεύει το περιεχόμενο του μιμούμενου (emulated) σκληρού δίσκου.

Χρησιμοποιείστε τις εντολές:

qemu-img create -f qcow aptosid.qcow 3G

Για να δημιουργήσει την εικόνα με όνομα αρχείου " aptosid.qcow. Η "3G" παράμετρος διευκρινίζει το μέγεθος του δίσκου - σε αυτήν την περίπτωση 3 GB. Μπορείτε να χρησιμοποιήσετε το επίθεμα Μ για τα μεγαμπάιτ (παραδείγματος χάριν "256M"). Δεν χρειάζεται να ανησυχείτε πάρα πολύ για το μέγεθος του δίσκου - το πρόγραμμα qcow συμπιέζει την εικόνα έτσι ώστε ο κενός χώρος δεν προστίθεται επάνω στο μέγεθος του αρχείου.

Εγκατάσταση του λειτουργικού συστήματος

Αυτό είναι την πρώτη φορά που θα πρέπει να ξεκινήσετε τον εξομοιωτή (emulator). Ένα πράγμα που πρέπει να λάβετε υπόψη: όταν κάνετε κλικ μέσα στο παράθυρο qemu, ο δείκτης του ποντικιού δεσμεύεται (grab). Για να τον απελευθερώσετε πατήστε:

Ctrl+Alt

Εάν πρέπει να χρησιμοποιήσετε μια εκκινήσιμη (bootable) δισκέτα, τρέξτε το Qemu με

qemu -floppy aptosid.iso -net nic -net user -m 512 -boot d aptosid.qcow

Εάν το CD-ROM σας είναι εκκινήσιμο, τρέξτε το Qemu με:

qemu -cdrom aptosid.iso -net nic -net user -m 512 -boot d aptosid.qcow

Τώρα εγκαταστήστε το aptosid σαν να επρόκειτο να το εγκαταστήσετε σε ένα πραγματικό σκληρό δίσκο

Τρέξιμο του συστήματος

Για να τρέξετε το σύστημά απλά γράψτε:

qemu [είκονα_σκληρού_δίσκου]

Μια καλή ιδέα είναι να χρησιμοποιηθούν επικαλυπτόμενες εικόνες. Με αυτόν τον τρόπο μπορείτε να δημιουργήσετε την εικόνα σκληρού δίσκου μια φορά και να πείτε του Qemu να αποθηκεύει τις αλλαγές σε ένα εξωτερικό αρχείο. Ξεφορτώνεστε όλη την αστάθεια, επειδή είναι τόσο εύκολο να επανέλθει στην προηγούμενη κατάσταση συστήματος (system state).

Για να δημιουργήσετε μια καλυπτόμενη εικόνα, γράψτε:

qemu-img create -b [[base''image]] -f qcow [[overlay''image]]

Αντικαταστήστε την εικόνα του σκληρού δίσκου με την βασική (base_image) εικόνα (στην περίπτωσή μας είναι το aptosid.qcow). Μετά από αυτό μπορείτε να ξεκινήσετε την εφαρμογή Qemu με :

qemu [επικαλυπτόμενη_εικόνα]

Η αρχική εικόνα θα αφεθεί άθικτη. Ένα εμπόδιο, η βασική εικόνα δεν μπορεί να μετονομαστεί ή να μετακινηθεί, η διαδικασία της επικάλυψης θυμάται το μονοπάτι της βασικής εικόνας στον δίσκο.

Χρησιμοποιώντας οποιαδήποτε πραγματική κατάτμηση ως ενιαία αρχική κατάτμηση μιας εικόνας σκληρού δίσκου

Μερικές φορές, μπορεί να επιθυμήτε να χρησιμοποιήσετε μια από της κατατμήσεις του συστήματος σας μέσα από το qemu (για παράδειγμα, εάν επιθυμείτε την έναρξη της πραγματικής σας μηχανής ή του qemu χρησιμοποιώντας μια δεδομένη κατάτμηση σαν διαχειριστής-root). Μπορείτε να το κάνετε αυτό χρησιμοποιώντας το λογισμικό Raid στο γραμμικό τρόπο (χρειάζεστε τον οδηγό πυρήνα linear.ko) και μια συσκευή loopback: το τέχνασμα είναι να κάνετε δυναμικά prepend ένα master boot record (MBR) στην πραγματική κατάτμηση που επιθυμείτε να ενσωματώσετε μέσα σε μια είκονα σκληρού δίσκου qemu.

Υποθέστε ότι έχετε μια απλή, unmounted κατάτμηση /dev/sdaN με κάποιο σύστημα αρχείων (filesystem) και εσείς επιθυμείτε να την κάνετε μέρος μιας εικόνας δίσκου qemu. Κατ' αρχάς, δημιουργείστε κάποιο μικρό αρχείο για να κρατήσει το MBR:

dd if=/dev/zero of=/path/to/mbr count=32

Εδώ, ένα αρχείο 16 KB (32 * 512 bytes) δημιουργείται. Είναι σημαντικό να μην είναι πάρα πολύ μικρό (ακόμα κι αν το MBR χρειάζεται μόνο ένα κομμάτι των 512 bytes), δεδομένου ότι όσο μικρότερο είναι, τόσο μικρότερο θα πρέπει να είναι το μέγεθος του λογισμικού της Raid συσκευής πράγμα που μπορεί να έχει αρνητική επίδραση στην απόδοση. Κατόπιν, ρυθμίζετε μια συσκευή loopback στο MBR:

losetup -f /path/to/mbr

Ας υποθέσουμε ότι η συσκευή είναι το /dev/loop0, γιατί εμείς δεν έχουμε χρησιμοποιήσει άλλα loopbacks. Το επόμενο βήμα είναι να δημιουργηθεί το "merged" MBR + /dev/sdaN εικονικό αρχείο χρησιμοποιώντας το Raid λογισμικό:

 modprobe linear
 mdadm --build --verbose /dev/md0 --chunk=16 --level=linear --raid-devices=2 /dev/loop0 /dev/sdaN

Η κατάληξη /dev/md0 είναι το τί θα χρησιμοποιήσετε ως μια qemu ακατέργαστη εικόνα δίσκου (μην ξεχνάτε να θέσετε τα δικαιώματα έτσι ώστε ο εξομοιωτής να μπορεί να έχει πρόσβαση σ'αυτό). Το τελευταίο (και κάπως δυσνόητο) βήμα είναι να τεθεί η διαμόρφωση δίσκων (γεωμετρία δίσκων και πίνακας κατατμήσεων) έτσι ώστε το αρχικό σημείο έναρξης χωρισμάτων στο MBR να ταιριάζει με ένα από το /dev/sdaN μέσα στο /dev/md0 (ένα offset ακριβώς 16 * 512 = 16384 bytes σε αυτό το παράδειγμα). Κάνετε αυτό χρησιμοποιώντας το fdisk στη μηχανή οικοδεσπότη όχι στον εξομοιωτή: η προεπιλεγμένη ρουτίνα ανίχνευσης δίσκων από το qemu οδηγεί συχνά σε μη roundable (δεν στρογγυλοποιούνται) kilobyte offsets (όπως 31.5 KB, όπως στο προηγούμενο τμήμα) που δεν μπορούν να ρυθμιστούν από τον κώδικα του Raid λογισμικού. Ως εκ τούτου, από τον οικοδεσπότη:

 fdisk /dev/md0

Δημιουργεί μία εννιαία αρχική κατάτμησή που αντιστοιχεί στο /dev/sdaN, και παίζει με το 's'ector από τις 'x'pert επιλογές μέχρι τον πρώτο κύλινδρο (όπου οι πρώτες ενάρξεις κατατμήσεων), αντιστοιχίες στο μέγεθος του MBR. Τέλος, ' w'rite το αποτέλεσμα στο αρχείο που κάνατε. Τώρα έχετε μία κατάτμηση που μπορείτε να τοποθετήσετε (mount) άμεσα από τον host σας, καθώς επίσης και μέρος μιας εικόνας δίσκου qemu:

 qemu -hdc /dev/md0 [...]

Μπορείτε φυσικά ακίνδυνα να θέσετε οποιοδήποτε bootloader σε αυτήν την εικόνα δίσκου χρησιμοποιώντας το qemu, υπό τον όρο ότι το αρχικό χώρισμα του /boot/sdaN περιέχει τα απαραίτητα εργαλεία.
